#### CSS anchored fallback container queries
<!--

#### CSS queries for anchored fallback container

Introduced `@container anchored(fallback)` to style descendants of anchor-positioned elements,
based on which of `position-try-fallbacks` is applied.

Such queries can be used to, for example, style an anchored element's tether or animations, based on how the anchor and the anchored element are positioned relative to each other.

  container-type: anchored;

  #anchored > .arrow {
    --arrow-rotation: 180deg;
  }
}

* Multilingual content.

## SVG features

Below are the new Scalable Vector Graphics (SVG) features in Microsoft Edge.


<!-- ------------------------------ -->
#### Enhanced support for presentation attributes on `<use>` and `<symbol>` elements<!-- css-related svg -->

The `<symbol>` element now supports `width`, `height`, `x`, and `y` as presentation attributes, giving you better control over layout.

The `<use>` element now supports `width` and `height` as presentation attributes.  This ensures consistent rendering of the `<use>` element when using `width` and `height` via both CSS and presentation attributes.

Previously, only CSS-based `width` and `height` attributes on the `<use>` element were supported; presentation attributes were ignored or inconsistent.

These changes ensure consistent rendering across CSS and markup.

Before:

```html
<svg width="100" height="100">
  <defs>
    <symbol id="sym">
      <rect width="100" height="100" fill="green"/>
    </symbol>
  </defs>
  <use href="#sym" style="width:40; height:40"/>
</svg>
```

Now:

```html
<svg width="100" height="100">
  <defs>
    <symbol id="sym" width="80" height="80" x="10" y="10">
      <rect width="100" height="100" fill="green"/>
    </symbol>
  </defs>
  <use href="#sym" width="40" height="40"/>
</svg>
```

## Web APIs

Below are the new Web API features in Microsoft Edge.


<!-- ------------------------------ -->
#### Allow more characters in JavaScript DOM APIs

The validation of the JavaScript DOM APIs has been relaxed to match the HTML parser.

Previous versions of the HTML parser allowed elements and attributes to have a wide variety of valid characters and names, but the JavaScript DOM APIs to create the same elements and attributes were more strict and didn't match the parser.

All of the previously allowed element/attribute names are still allowed with the new behavior.  This avoids compatibility issues.

See also:
* [Valid/Invalid characters in document.createElement()](https://github.com/whatwg/dom/issues/849)
* []


<!-- ------------------------------ -->
#### The `clipboardchange` event

The `clipboardchange` event fires whenever the system clipboard contents are changed either by a web app or any other system application.  This allows web apps such as remote desktop clients to keep their clipboards synchronized with the system clipboard.

Listening for this event provides an efficient alternative to polling the clipboard (using JavaScript) for changes.

#### `dataTransfer` property for `insertFromPaste`, `insertFromDrop`, and `insertReplacementText` input events

<!-- the DataTransfer object is cap'd: https://developer.mozilla.org/en-US/docs/Web/API/DataTransfer -->

Populated the `dataTransfer` property on input events with `inputType` of `insertFromPaste`, `insertFromDrop`, and `insertReplacementText`, to provide access to clipboard and drag-and-drop data during editing operations in `contenteditable` elements.

The `dataTransfer` object contains the same data that was available during the `beforeinput`<!-- todo: link to Ref, to prove name --> event.

This feature only applies to `contenteditable` elements.  For form controls (such as `textarea` and `input`), the behavior remains unchanged: the `data` property contains the inserted text, and `dataTransfer` remains `null`.

Safari and Firefox already support this functionality.  Chromium adopting this feature enhances interoperability across browsers, providing a more consistent experience for web developers.

#### FedCM-support structured JSON responses from identity providers

Federated Credential Management (FedCM): FedCM-support structured JSON responses from identity providers (IdPs) allows identity providers to return structured JSON objects instead of plain strings to relying parties (RPs) via the `id_assertion_endpoint`.

This change eliminates the need to manually serialize and parse JSON strings, simplifying integration.

This change enables more dynamic and flexible authentication flows, allowing relying parties to interpret complex responses directly, and thus support varied protocols (such as OAuth2, OIDC, or IndieAuth) without out-of-band agreements.

#### IDNA `ContextJ` rules<!-- todo: clarify, make relatable -->
<!--

IDNA<!-- todo: expand IDNA, eg Internationalizing Domain Names in Applications https://www.bing.com/search?q=IDNA+rules+for+URLs --> is the mechanism for non-ASCII characters in domain names, encoding a URL like `http://네이버.한국/` as `http://xn--950bt9s8xi.xn--3e0b707e/` (that gets redirected to `https://www.naver.com`<!-- todo: non-ms url ok? -->).

The processing is defined by [4 Processing](https://www.unicode.org/reports/tr46/#Processing) in _Unicode IDNA Compatibility Processing_, and is invoked by [3.3. IDNA](https://url.spec.whatwg.org/#idna) in the _URL Standard_.

The URL spec sets the `CheckJoiners` flag, which enables the `ContextJ` rules in IDNA2008:
* [Appendix A.1.  ZERO WIDTH NON-JOINER](https://www.rfc-editor.org/rfc/rfc5892.html#appendix-A.1) in _The Unicode Code Points and Internationalized Domain Names for Applications (IDNA)_.
* [Appendix A.2.  ZERO WIDTH JOINER](https://www.rfc-editor.org/rfc/rfc5892.html#appendix-A.2) _The Unicode Code Points and Internationalized Domain Names for Applications (IDNA)_.

This disallows ZWNJ (U+200C ZERO WIDTH NON-JOINER) and ZWJ (U+200D ZERO WIDTH JOINER) in most places in URLs.

The implementation is to simply pass the UIDNA_CHECK_CONTEXTJ option to ICU, where this rule is implemented:
* [isLabelOkContextJ()](https://source.chromium.org/chromium/chromium/src/+/main:third_party/icu/source/common/uts46.cpp;l=1137-1204;drc=8a1988938d4298fbe8fb499b1a59fe4b04a21b15)

This would fix over 200 subtests in WPT relating to IDNA, which already pass in Firefox and Safari:
* [Enable IDNA ContextJ rules by default](https://chromium-review.googlesource.com/c/chromium/src/+/6990929)
* [web-platform-tests dashboard](https://wpt.fyi/results/url/IdnaTestV2.window.html)

All of the IdnaTestV2 cases that would regress from shipping ICU 77 would also be fixed again by this change.  See [Feature: ICU 77 (supporting Unicode 16)](https://chromestatus.com/feature/5143313833000960).

#### Origin-bound cookies (by default)

In Chrome<!-- todo: browser --> 143, cookies are bound to their setting origin (by default) such that they're only accessible by that origin, such as:
* Sent on a request.
* Visible through `document.cookie`.

Cookies may ease the host and port binding restrictions through use of the `Domain` attribute, but all cookies are bound to their setting scheme.

To revert this change, the temporary enterprise policies `LegacyCookieScopeEnabled` and `LegacyCookieScopeEnabledForDomainList` are available.  These policies will stop working in Chrome<!-- todo: browser --> 150.

#### Web Smart Card API

The Web Smart Card API enables smart card (PC/SC) applications to move to the Web platform.

The Web Smart Card API gives smart card (PC/SC) applications access to the PC/SC implementation (and card reader drivers) that's available in the host operating system (OS).

Admins can control the availability of the Web Smart Card API at either scope:
* Globally, by using the `DefaultSmartCardConnectSetting` policy.
* Per-application, by using the `SmartCardConnectAllowedForUrls` and `SmartCardConnectBlockedForUrls` policies.

#### `WebTransport` application protocol negotiation

`WebTransport` application protocol negotiation allows negotiation of the protocol used by the web application within the `WebTransport` handshake.

A web application can specify a list of application protocols that are offered when constructing a `WebTransport` object, which are then conveyed to the server via HTTP headers.

If the server picks one of those protocols, the server can indicate that within response headers, and that reply is available within the `WebTransport` object.

###### Deprecating `savedTabGroups` as an individual value in `SyncTypesListDisabled`

To align desktop behavior with mobile and simplify sync management, the individual `savedTabGroups` datatype will be deprecated, and will no longer be an individually customizable value within the `SyncTypesListDisabled` enterprise policy.

Starting with Chrome<!-- todo: Edge, the Chromium engine --> 142, if your `SyncTypesListDisabled` policy disables either `tabs` or `savedTabGroups`, both data types will now be considered disabled.

This means that disabling `tabs` will also disable saved tab groups, and vice-versa.

The `savedTabGroups` value will be entirely removed from the list of supported datatypes for the `SyncTypesListDisabled` policy.

Previous policy behavior:
* The `SyncTypesListDisabled` enterprise policy allowed administrators to specifically disable the synchronization of `savedTabGroups` on desktop platforms.
* On mobile platforms, the synchronization of Tab Groups functionality is managed by the `tabs` datatype.

**Action required by administrators:**

For admins who have saved tab groups disabled, and intend to keep this behavior, make sure to explicitly disable the `tabs` datatype.  This guarantees the desired behavior before the `savedTabGroups` value is fully removed.

#### Extended lifetime shared workers

Expires on April 21, 2026.

Extended-lifetime shared workers introduces the `extendedLifetime` option for the `SharedWorker` constructor.

This feature allows a shared worker to stay alive for a short period after all connected clients (such as pages) have been unloaded.  This is useful for performing asynchronous tasks that need to complete after a page unloads, such as saving data to IndexedDB or sending analytics.

Without this feature, those tasks require a Service Worker, which can be too heavyweight for this use-case.  This origin trial provides a lightweight alternative.
